## 1. 배경 및 목표

### 1.1 문제

사용자가 이사·여행 등으로 **지역별 분리수거 규칙이 다른 상황**에서, 급하게 버리려 할 때 “어디에 무엇을 어떻게” 확신하기 어렵다.

### 1.2 제품 비전

**실시간 카메라 인식 + 지자체 맞춤 규칙 + 단계별 픽토그램 가이드**를 한 화면에서 제공하여, 텍스트 의존도를 낮추고 신뢰 가능한 로컬 정보로 배출 행동을 유도한다.

### 1.3 목표(Goals)

- GPS(또는 사용자 선택 지역) 기준으로 **현재 행정구역의 배출 규칙**을 노출한다.
- 카메라로 물체를 비추면 **YOLO 기반 실시간 검출**과 함께 분류 힌트를 제공한다.
- 인식이 불확실하거나 사용자가 결과를 거부하면 **대화형 클리어**(질문·선택지·텍스트 또는 음성)로 무엇을 버릴지 좁히고 **로컬 DB 품목과 매칭**하여 최종 가이드를 제시한다.
- 하단 슬라이딩 카드에 **분류 결과·지역 팁·액션 단계·배출 요일**을 한 번에 보여준다.
- 음성·고화질 촬영으로 **예외 상황(인식 실패, 세부 질문)**을 보완한다.

### 1.4 비목표(Non-Goals)

- 전국 모든 지자체 데이터를 첫 버전에서 완비하는 것(단, 데이터 소스·확장 구조는 명시).
- 일반 쓰레기 적발·신고 등 행정 업무 시스템과의 직접 연동(명시적 요구 없음).

## 2. 사용자 및 시나리오

### 2.1 핵심 페르소나

`problem.md`에 정의된 **고양시 일산 지역으로 이사 온 60대 사용자** 등: 스마트폰 기본 사용 가능, **급한 상황·큰 글자·직관적 그림** 선호.

### 2.2 주요 시나리오

1. 분리수거함 앞에서 페트병을 들고 앱 실행 → 위치 확인 → 카메라로 비춤 → 박스·라벨로 인식 → 카드에서 “비우기·라벨 제거·압착” 확인 후 배출.
2. 실시간 인식이 불안정 → **촬영 버튼**으로 고화질 분석(Gemini) 요청.
3. “뚜껑이 안 빠져요” 같은 세부 질문 → **마이크**로 음성 입력 후 안내(후속 답변 UX는 구현 단계에서 구체화).
4. 모델이 **확실하지 않다**(낮은 일치도, 다중 객체 경쟁, 사용자 「맞아요/아니에요」 거부). → 바텀시트에 **안내 후보**와 함께 **대화 패널**이 열리고, 선택지 또는 자유 입력으로 재질·용도 등을 거쳐 **확정 품목** 하나를 고른 뒤 **기존과 동일한 로컬 규칙 카드**로 이어진다.

### 3.5 ⑤ 불확실 인식 시 — 대화형 품목 확정(Clarification / Dialogue)

실시간·정밀 비전 결과가 있어도 **확신이 없을 때** 사용자와 짧게 주고받아 **품목 레코드를 하나로 수렴**한 뒤, 기존 흐름(로컬 규칙 + 카드)을 동일하게 적용한다.

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| C-1 | **진입 조건**(구현 시 임계값 확정): 모델 **신뢰도 미만**, 상위 후보 간 **점수 차 과소**, **다중 검출**, 사용자의 **「이 물건이 아니에요」**/재촬영 반복 등. | P0 |
| C-2 | 바텀시트 또는 **대화 패널**에 **메시지 스레드**(가장 최근 턴만 강조해도 됨): 앱 질문 1턴 + 사용자 응답. | P0 |
| C-3 | 우선 **객관식·빠른 선택지**(재질, 용도, 포장/용기 종류 등) 제공. 자유 입력(텍스트)은 같은 패널에서 지원. | P0 |
| C-4 | 음성은 **선택 기능**(STT 후 동일 패널에 반영). 오프라인·비용 때문에 MVP는 텍스트+선택지 우선 가능. | P1 |
| C-5 | 대화 상태에서 **마지막 캡처 프레임(또는 썸네일)**을 참고 맥락으로 표시 가능(저장은 프라이버시 정책 준수). | P1 |
| C-6 | 수렴된 품목은 **품목사전/앱 DB 키**(`ni_idx`, `item_id` 등)와 연결되어 **카드 요약(B-2~B-5)** 과 동일한 데이터 소스에서 표시된다. DB에 없으면 안전 폴백 문구 및 외부 링크(선택). | P0 |
| C-7 | 대화 종료 후 **확정 품목명·출처 크레딧** 표시 및 **다시 시작**으로 재진입. | P0 |

## 4. UX / UI 스펙

### 4.1 레이아웃(세로 스택)

1. **상단**: 슬림 위치 바 + GPS + 설정.
2. **중앙**: 카메라 뷰 + Bounding Box + 플로팅 라벨.
3. **하단**: 슬라이딩 정보 카드.
4. **오버레이**: 마이크·촬영 등 플로팅 버튼(카메라와 겹치지 않게 안전 영역 규칙 적용).
5. **(조건부)** 불확실 인식 시: 카드 확장 또는 **대화 패널**(C-2)이 카드 상단 또는 별도 바텀시트로 노출된다.

### 4.2 디자인 토큰(브랜드)

| 토큰 | 값 | 용도 |
|------|-----|------|
| 포레스트 그린 | `#2D5A27` | 주색, 환경·청결 연상 |
| 소프트 화이트 | `#F8F9FA` | 배경·카드 베이스 |
| 보조 | 글래스모피즘용 반투명 레이어, 소프트 그린 틴트 | 헤더·카드 |

### 4.3 타이포그래피

- **모던 산세리프**, 고해상도(4K 에셋·벡터 UI 대응)에서도 선명한 계층: **제목 Bold**, 본문·크레딧은 상대적으로 작되 **대비(Accessibility)** 기준 준수.
- “급한 상황” 전제로 **픽토그램 우선**, 텍스트는 최소·핵심만.

### 4.4 스타일 키워드

- **글래스모피즘**(블러·얇은 테두리·층위감) — 안드로이드 퍼스트, iOS에서는 동일 가이드로 맞춤.
- Bounding Box: **슬릭한 네온 글로우**.
- 전체:**깔끔하고 사용자 친화적**, **고해상도 UI** 에셋 대응.

### 4.5 접근성(추가 권장)

- 시스템 글자 크기 존중, 터치 타겟 최소 크기 준수.
- 색만으로 정보를 전달하지 않기(아이콘·패턴 병행).

## 5. 콘텐츠 및 데이터

| 항목 | 설명 |
|------|------|
| 지자체 규칙 | 행정구역 코드 ↔ 배출 분류·문구·팁 매핑; **공식 출처 명시**(예: 고양시청). |
| 배출 요일 | 지역별 수거 요일/종류 캘린더 데이터 및 갱신 주기 정의. |
| 모델 레이블 | YOLO 클래스와 앱 내 카테고리·지역 규칙 키 간 매핑 테이블. |
| 대화 수렴 | 후보 라벨·사용자 응답을 바탕으로 **품목 레코드를 선정**(규칙 기반 순위 또는 LLM **그라운딩**을 `app_search_keyword`, `app_item_rule` 등에 제한해 사용). 세션 종료 후 최종 선택만 카드 바인딩에 사용. |

## 6. 기술 고려사항(개발 핸드오프)

| 영역 | 내용 |
|------|------|
| 비전 | 카메라 프레임 → **YOLO26** 추론 → 박스·라벨 좌표 UI 오버레이. |
| 정밀 분석 | 캡처 이미지 → **Gemini**(멀티모달) 분석 API — 레이트 리밋·비용·프라이버시 동의. |
| 대화형 확정 | 낮은 신뢰도 시 서버 또는 온디바이스에서 **스트리밍 채널**(텍스트 최소)·**후보 리스트 줄이기**·최종 `item_id` 결정. LLM 사용 시 출력은 허용된 DB 행 참조 또는 구조화 JSON으로 받아 검증 후만 카드 렌더. |
| 위치 | Android Fused Location / iOS Core Location + 수동 선택 폴백. |
| 음성 | 온디바이스 또는 클라우드 STT + 질의 응답 파이프라인. |

## 7. 성공 지표(초안)

- 인식 성공 후 **카드 가이드 확인까지** 이탈률 감소.
- 동일 세션 내 **재촬영·재질문 횟수**(불확실성 지표).
- **대화 패널 종료 후** 사용자가 선택한 최종 품목 기준 카드까지 도달 비율(대화 세션 성공률).
- 사용자 피드백 기준 **“지역 규칙이 도움이 되었다”** 긍정률(설문·인앱 1탭 등).

## 8. 리스크 및 미결정 사항

- **YOLO26** 실제 모델명·라이선스·온디바이스 vs 서버 추론 분리.
- Gemini 호출 시 **개인정보·이미지 보관 정책** 및 동의 문구.
- 다중 객체·부분 가림 시 **어떤 객체를 기준으로 카드를 채울지** UX 확정.
- 대화 수렴에서 **LLM 환각** 방지(LLM 결과를 DB 존재 여부와 교차 검증 없이 카드에 반영 금지).
- `problem.md`의 **일산서구** 등 실제 타깃 행정구역과 화면 예시 **일산동구** 명칭 통일 여부.

## 9. 마일스톤 제안

1. **MVP**: H/M/B 핵심 + 촬영→Gemini 폴백 + 단일 지자체 시드 데이터 + **신뢰도 낮음 시 선택지 중심 대화 패널(C-1~C-3,C-6,C-7)**.
2. **v1.1**: 음성 기반 입력(C-4) 강화, 수동 지역 고도화, 배출 캘린더 연동 강화.
3. **v1.2**: 지자체 확장, 오프라인 캐시, 접근성 강화.
